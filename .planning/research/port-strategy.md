# Port Strategy: DGC MiniApp Design System → shadcn Registry

**Researched:** 2026-04-24
**Mode:** Feasibility + Comparison
**Verdict:** YES — port is feasible and straightforward. Existing token model aligns well with shadcn conventions; JSX kit maps cleanly to Radix primitives. Main work is token transformation (HEX → HSL), Tailwind config generation, and `registry-item.json` authoring per component.
**Confidence:** HIGH (shadcn conventions), MEDIUM (specific Radix mappings per specimen).

---

## 1. Tailwind vs plain CSS vars

**Decision:** Dual layer. Keep `--blue-900`, `--space-4`, `--radius-md`, `--text-body` as canonical tokens in `globals.css`. Add a Tailwind `theme.extend` that references them so shadcn components can use `bg-accent`, `text-foreground`, `rounded-md`, `p-4` idiomatically.

**Why:** shadcn's `tailwind.cssVariables: true` mode is the default. Components generated by the shadcn CLI expect class names like `bg-primary text-primary-foreground` to resolve via `hsl(var(--primary))`. Preserving the existing `--blue-*` and `--space-N` ramps avoids a rewrite of the guideline CSS and keeps the builder app compatible. Add a semantic alias layer on top:

```css
:root {
  /* Keep existing scale tokens */
  --blue-900: 214 85% 34%;  /* was #0D47A1 as HSL channels */
  --space-4: 16px;

  /* Add shadcn semantic aliases */
  --primary: var(--blue-900);
  --primary-foreground: 0 0% 100%;
  --background: 0 0% 100%;
  --foreground: 214 30% 12%;
  --accent: 214 85% 34%;
  --border: 214 15% 88%;
  --ring: 214 85% 34%;
  --radius: 0.75rem; /* = --radius-md 12px */
}
```

```ts
// tailwind.config.ts
theme: {
  extend: {
    colors: {
      primary: { DEFAULT: "hsl(var(--primary))", foreground: "hsl(var(--primary-foreground))" },
      accent:  { DEFAULT: "hsl(var(--accent))",  foreground: "hsl(var(--accent-foreground))" },
      border:  "hsl(var(--border))",
      ring:    "hsl(var(--ring))",
    },
    borderRadius: { lg: "var(--radius)", md: "calc(var(--radius) - 2px)", sm: "calc(var(--radius) - 4px)" },
    spacing: { /* 4pt scale mirrored from --space-1..9 */ },
    fontFamily: {
      latin: ["Inter", "IBM Plex Sans", "Roboto", "system-ui", "sans-serif"],
      khmer: ["Noto Sans Khmer", "Kantumruy Pro", "Battambang", "sans-serif"],
      sans:  ["Inter", "Noto Sans Khmer", "system-ui", "sans-serif"], // bilingual default
      mono:  ["JetBrains Mono", "monospace"],
    },
  },
}
```

## 2. HSL vs HEX

**Decision:** Convert all color tokens to **HSL channel-triplet** format (`214 85% 34%`) — no `hsl()` wrapper, no commas. This is shadcn's Tailwind v3 convention; Tailwind v4 wraps at definition time (see §3 below).

**Why:** Radix state modifiers like `data-[state=open]:bg-primary/90` require alpha-at-callsite syntax, which only works if the var is a bare triplet wrapped by `hsl(var(--x) / <alpha-value>)` in the Tailwind config.

**Tooling:** Write a one-shot Node script `scripts/hex-to-hsl.mjs` that parses `project/colors_and_type.css`, converts every `#rrggbb` to channel triplets, and emits `src/styles/tokens.css`. ~30 colors — trivial. Reject: culori, colorjs.io for runtime — we only need build-time conversion. The builder app continues to store HEX in its editor UI and converts on export.

**Caveat:** If targeting Tailwind v4 from day one, switch to `--primary: hsl(214 85% 34%)` at declaration and reference directly in `@theme inline` without wrapping. Recommend **Tailwind v3.4** for the initial port (mature ecosystem, shadcn CLI defaults align), plan v4 migration as a later milestone.

## 3. Dark mode strategy

**Decision:** Migrate `[data-theme="dark"]` → `.dark` class on `<html>`. Scope overrides in a `.dark { --background: ...; --foreground: ...; }` block. Keep the builder app's `#themeToggle` but have it toggle `.dark` instead. Update `builder/app.js` in a follow-up patch (non-blocking).

**Why:** shadcn's `next-themes` integration, Radix `data-state` selectors, and Tailwind's `dark:` variant all assume `.dark` on the root. `[data-theme]` works but requires a custom `darkMode: ['class', '[data-theme="dark"]']` Tailwind selector array — fragile and non-standard for consumers.

## 4. Mobile-first + Khmer fonts

**Decision:** Declare bilingual font stack at the Tailwind `fontFamily.sans` level so every shadcn component inherits Khmer fallback automatically. No per-component font switching needed.

```ts
fontFamily: {
  sans: ["var(--font-latin)", "var(--font-khmer)", "system-ui", "sans-serif"],
  khmer: ["var(--font-khmer)", "sans-serif"],  // use for Khmer-only blocks
}
```

Where `--font-latin: "Inter", "IBM Plex Sans", "Roboto"` and `--font-khmer: "Noto Sans Khmer", "Kantumruy Pro", "Battambang"`. Font faces load via `@font-face` in `src/styles/fonts.css` (ported from `project/fonts/fonts.css`) or via `next/font` if on Next.js.

**Khmer line-height:** Add a `.font-khmer` utility with `line-height: 1.6` in a Tailwind plugin, or use shadcn's `cn()` with a `leading-khmer` class. Keep the +1px Khmer size bump as a Typography primitive prop: `<Text lang="km" bumpSize />`.

Mobile-first: shadcn defaults are mobile-first; no changes needed. The existing `.is-mobile` preview constraint becomes a Storybook viewport story.

## 6. Radix vs headless-ui

**Decision:** Radix across the board. shadcn is built on Radix; headless-ui is a React/Vue-only library with weaker coverage (no Toggle Group, no Navigation Menu, no Scroll Area). Radix gives us Dialog, Popover, Tabs, Toggle, Switch, Checkbox, Radio, Select, Tooltip, Dropdown, ScrollArea, Toast — all needed.

**Exception:** Use `sonner` for toast notifications. It has better DX than `@radix-ui/react-toast` and is the shadcn default since 2024.

## 7. Icons

**Keep Lucide.** Already used via CDN in the builder; shadcn defaults to Lucide React. Install `lucide-react`. The `project/ICONOGRAPHY.md` mapping carries over 1:1. For tree-shaking, import per-icon: `import { FileText } from "lucide-react"`.

## 8. What CANNOT be directly ported

Flag these as **out of scope for the shadcn registry** — they belong to the SuperApp host, not MiniApp consumers:

- **`SuperAppZone` / `MiniHeader`** (from `components.jsx`) — duplicate SuperApp chrome. MiniApps should never render their own SuperApp header; the host provides it. Porting these would encourage consumers to build broken double-header layouts.
- **`superapp-host.html`, `superapp-logo.html`** specimens — demonstrate host-level composition, not reusable UI. Keep as documentation (`docs/host-integration.mdx`) only.
- **Builder app (`builder/`)** — vanilla editor, not a registry item. Keep as-is; optionally port to a Next.js route in a separate milestone.
- **`flag-brand.html`** — brand asset usage guidance, ship as `registry:lib` with SVG imports, not as a component.
- **`content-tone.html`, `type-bilingual.html`, `type-khmer.html`, `type-latin.html`, `colors-*`, `spacing.html`, `radii.html`, `elevation.html`, `iconography.html`** — pure documentation specimens. Render as MDX pages in a docs site; not registry items.

**Partial ports with caveats:**
- `Hero` JSX currently hardcodes SuperApp brand copy — port the layout primitive but make all copy/CTA props required.
- `app-header.html` is 3-slot (drawer + title + 2 actions + badges). Port with named slots (`<AppHeader left={...} right={...}>`) rather than rigid props.

## Recommended phase structure (for roadmap)

1. **Phase 1 — Foundations** (1 registry:theme + 1 registry:font + 1 registry:lib): HEX→HSL conversion script, tokens.css, tailwind.config.ts, font stack, cn() utility. Ship as `theme-dgc-miniapp`.
2. **Phase 2 — Primitives** (8 registry:ui): button, input, textarea, checkbox, radio-group, switch, select, alert. Covers ~80% of specimens; validates the registry pipeline.
3. **Phase 3 — Overlays** (5 registry:ui): dialog, sheet, tabs, tooltip, combobox, toast/sonner. Radix-heavy; needs portal + focus-trap validation.
4. **Phase 4 — MiniApp blocks** (7 registry:block): service-tile, service-grid, list-row, document-row, file-row, file-uploader, hero-banner, app-header, doc-viewer, step-indicator, empty-state, nav-row, section-header, segmented, timestamp.
5. **Phase 5 — Docs site** (Next.js + MDX): migrate the 9 documentation specimens into MDX pages; embed live examples.
6. **Phase 6 — Builder port (optional)**: rebuild `builder/` as a Next.js route using the new registry components.

**Research flags:**
- Phase 2: Validate 48px touch-target + `--shadow-focus` ring translate cleanly to Tailwind `h-12 focus-visible:ring-[3px] ring-ring/25`.
- Phase 3: Khmer text inside Radix portals needs `lang="km"` attribute propagation — verify.
- Phase 5: Decide Nextra vs Fumadocs vs shadcn's own `ui.shadcn.com` template.
